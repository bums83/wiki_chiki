# PostgreSQL + VectorChord, часть 1

Конспект первой части статьи про локальную инфраструктуру гибридного поиска на базе **PostgreSQL + VectorChord**. Это не копия исходника, а сжатое изложение ключевых идей и архитектуры.

## Источник

- [Habr, часть 1: PostgreSQL + VectorChord = Гибридный поиск. Инфраструктура](https://habr.com/ru/articles/1024810/)
- [Habr, часть 2: PostgreSQL + VectorChord = Гибридный поиск. Безоблачная](https://habr.com/ru/articles/1024818/)

## О чем первая часть

Первая часть закрывает базовый фундамент:
- разворачивание PostgreSQL с расширениями VectorChord,
- описание таблиц для документов и чанков,
- подключение `VechordRegistry`,
- пайплайны CRUD для документов,
- базовый hybrid search,
- простой реранкинг без облачных API.

Смысл конструкции простой: хранить исходные документы отдельно, а поиск вести по чанкам, у которых есть:
- текст,
- полнотекстовое представление,
- embedding,
- связь с родительским документом.

## База и расширения

Автор использует Docker и образ:
- `tensorchord/vchord-suite:pg18-latest`

Ключевая деталь: расширения не ставятся руками в compose. Их потом поднимает сам `VechordRegistry` при первом `async with vr:`.

Это важно, потому что инфраструктура становится самоинициализируемой:
- `vchord`
- `vchord_bm25`
- `pg_tokenizer`

Также вводится `namespace`, чтобы таблицы создавались с префиксом и не конфликтовали между окружениями.

## Модель данных

Архитектура строится на трех таблицах:

### 1. BaseTable
Общая база для всех сущностей:
- `metadata`
- `created_at`
- `updated_at`

### 2. Document
Единица управления контентом:
- `uid`
- `title`
- `text`

Тут лежит исходный полный документ.

### 3. Chunk
Единица поиска:
- `uid`
- `doc_id`
- `content`
- `content_tsv`
- `embedding`
- `chunk_index`

Именно `Chunk` участвует в поиске.

## Почему документы и чанки разнесены

Это правильный паттерн для RAG и search-систем:
- документ слишком крупный для точного поиска,
- chunk удобен для retrieval,
- документ остается источником правды,
- chunk можно перестроить при изменении chunking strategy или embedding model.

## VechordRegistry как центр системы

`VechordRegistry` здесь выполняет сразу несколько ролей:
- подключение к PostgreSQL,
- установка нужных расширений,
- создание таблиц,
- CRUD-операции,
- keyword search,
- vector search,
- pipeline execution.

То есть это не просто thin wrapper, а основной operational layer вокруг базы.

## Пайплайн создания документа

При создании документа происходят два шага:
1. сохранить `Document`,
2. нарезать `text` на chunks и записать `Chunk`.

Для этого используется `VechordPipeline`.

Идея здравая:
- операция становится декларативной,
- шаги можно переиспользовать,
- легко вставить дополнительные стадии,
- меньше ручной glue-логики в сервисном коде.

В первой части фигурируют такие инструменты и компоненты:

## Чем чанкируют и чем эмбеддят в первой части

В первой части автор сознательно берет упрощенный стек:
- `SpacyChunker`
- `SpacyDenseEmbedding`

Это не production-grade вариант. Сам автор это прямо признает.

Но для первой итерации это нормальный ход, потому что позволяет:
- проверить всю цепочку end-to-end,
- не зависеть от облачных API,
- быстро собрать рабочий prototype.

## Обновление документа

Так как явного `update` в используемом abstraction layer автор не нашел, он делает практичный обходной путь:
- читает старый документ,
- собирает новый объект с тем же `uid` и `created_at`,
- удаляет старую запись,
- записывает новую,
- чанки пересобираются заново.

За счет `ForeignKey` с каскадным удалением старые чанки исчезают автоматически.

Это грубовато, но логично. Для раннего этапа системы подход рабочий.

## Поиск

В первой части есть три режима:

### Keyword search
Ищет по `content_tsv`.

### Vector search
Ищет по embedding-вектору чанка.

### Hybrid search
Объединяет keyword и vector retrieval, чтобы повысить recall.

Главная идея hybrid search:
- keyword лучше ловит точные совпадения,
- vector лучше ловит смысловую близость,
- вместе они уменьшают вероятность пропустить нужный фрагмент.

## Реранкинг

Для первой части автор использует **Reciprocal Rank Fusion (RRF)**.

Почему это разумно:
- не нужны внешние API,
- не нужна отдельная нейросеть,
- легко собрать поверх двух списков кандидатов,
- хорошо подходит как baseline.

Ограничение тоже очевидно:
- это не semantic reranker,
- он не понимает смысл пары `query -> document` так глубоко, как cross-encoder модель.

Именно поэтому во второй части автор уже уходит к локальному neural reranker.

## Что важно practically

Первая часть полезна не как “идеальный прод”, а как хороший skeleton:
- Postgres как единое хранилище,
- векторный и keyword поиск в одной системе,
- registry/pipeline-подход,
- отделение документа от chunk-слоя,
- возможность потом заменить chunker, embedder и reranker без слома общей архитектуры.

Это и есть сильная сторона конструкции.

## Ограничения первой части

Слабые места первой части:
- `spacy`-эмбеддинги как временная заглушка,
- chunking еще слишком примитивный,
- update реализован через remove + recreate,
- reranking пока baseline-уровня,
- нет серьезной оценки качества retrieval.

Но как стартовая оффлайн-схема это нормальная база.

## Связь со второй частью

Во второй части этот фундамент усиливается:
- локальный Jina embedding через `llama.cpp`,
- локальный reranker через `transformers + PyTorch`,
- более сильный semantic chunker,
- попытка перейти от demo к более production-like стеку.

## Вывод

Первая часть показывает правильную последовательность:
1. сначала собрать простую, прозрачную инфраструктуру,
2. проверить пайплайны и модель данных,
3. только потом усиливать качество retrieval-модулей.

Это хороший инженерный ход. Сначала каркас, потом дорогие улучшения.
