# PostgreSQL + VectorChord

`PostgreSQL + VectorChord` здесь рассматривается как локальный стек для гибридного поиска, где в одной системе объединяются хранение документов, полнотекстовый поиск, векторный поиск, реранкинг и эволюция retrieval-контура от простого baseline к более сильной offline-архитектуре.

Это не пересказ двух статей по отдельности, а одна цельная инженерная схема: как собрать рабочий retrieval pipeline на Postgres, а потом усилить его без ухода в облачные API.

## Источники

- [Habr, часть 1: PostgreSQL + VectorChord = Гибридный поиск. Инфраструктура](https://habr.com/ru/articles/1024810/)
- [Habr, часть 2: PostgreSQL + VectorChord = Гибридный поиск. Безоблачная](https://habr.com/ru/articles/1024818/)

## Что это за архитектура

Базовая идея проста:
- исходный документ хранится целиком,
- для поиска он режется на чанки,
- по чанкам строятся keyword и vector retrieval,
- кандидаты дополнительно реранжатся,
- все ключевые компоненты остаются локальными.

Это хороший паттерн для RAG и search-систем, потому что документ и единица поиска не смешиваются. Полный текст остаётся источником правды, а чанки становятся рабочим слоем retrieval.

## Инфраструктурный фундамент

В основе лежит PostgreSQL с расширениями `vchord`, `vchord_bm25` и `pg_tokenizer`.

Автор использует Docker-образ `tensorchord/vchord-suite:pg18-latest`, а сами расширения поднимаются не вручную в compose, а через `VechordRegistry` при первом `async with vr:`.

Практически это даёт несколько полезных свойств:
- база самоинициализируется,
- расширения и токенизаторы создаются в одном operational path,
- таблицы можно неймспейсить через `namespace`,
- инфраструктура легче переносится между окружениями.

## Модель данных

Архитектура строится вокруг трёх сущностей.

### BaseTable
Общие поля:
- `metadata`
- `created_at`
- `updated_at`

### Document
Сущность полного документа:
- `uid`
- `title`
- `text`

### Chunk
Сущность retrieval-слоя:
- `uid`
- `doc_id`
- `content`
- `content_tsv`
- `embedding`
- `chunk_index`

Такое разделение корректно по нескольким причинам:
- документ слишком крупный для точного retrieval,
- chunk удобен для полнотекстового и векторного поиска,
- стратегию chunking и embeddings можно менять без смены всей модели хранения,
- обновление документа автоматически ведёт к пересборке retrieval-слоя.

## Роль VechordRegistry

`VechordRegistry` в этой схеме, это не просто коннектор к базе, а основной operational layer.

Он закрывает сразу несколько задач:
- подключение к PostgreSQL,
- инициализацию расширений,
- создание таблиц,
- CRUD-операции,
- keyword search,
- vector search,
- pipeline execution.

За счёт этого логика приложения получается компактнее. Вместо ручного glue-кода вокруг SQL возникает единый слой, который держит data model и retrieval operations вместе.

## Пайплайн работы с документами

Создание документа устроено как двухшаговый pipeline:
1. сохранить `Document`,
2. нарезать текст на чанки и сохранить `Chunk`.

Для этого используется `VechordPipeline`.

С инженерной точки зрения это хороший выбор, потому что:
- операция описывается декларативно,
- шаги можно переиспользовать,
- легко встраивать дополнительные стадии,
- сервисный код не превращается в длинную цепочку ручных действий.

Обновление документа реализовано через remove + recreate:
- старый документ читается,
- создаётся новая версия с тем же `uid` и `created_at`,
- старая запись удаляется,
- чанки пересобираются заново.

Это не самый изящный вариант, но для раннего слоя abstractions он рабочий. За счёт `ForeignKey` с каскадным удалением старые чанки исчезают автоматически.

## Первый retrieval-контур

В стартовой версии используются:
- `SpacyChunker`
- `SpacyDenseEmbedding`
- keyword search по `content_tsv`
- vector search по embedding-вектору
- hybrid search
- `Reciprocal Rank Fusion` как baseline-реранкер

Это не production-grade стек. Но как первый работающий skeleton он полезен.

Он позволяет быстро проверить:
- data model,
- пайплайны записи и обновления,
- работу полнотекстового поиска,
- векторный retrieval,
- механику hybrid search.

## Почему hybrid search здесь уместен

Смысл hybrid search в том, чтобы объединить сильные стороны двух подходов:
- keyword search лучше ловит точные совпадения,
- vector search лучше ловит смысловую близость.

Вместе они повышают recall и уменьшают вероятность пропустить нужный фрагмент.

Для первой итерации этого достаточно. Но точность ранжирования остаётся ограниченной, потому что RRF лишь объединяет списки кандидатов и не делает глубокую semantic оценку пары `query -> chunk`.

## Усиление quality-layer без облаков

Во второй итерации архитектура остаётся прежней по форме, но усиливается по качеству:
- embedding заменяется на `jina-embeddings-v4-text-retrieval`,
- RRF заменяется на локальный neural reranker `jina-reranker-v3`,
- `SpacyChunker` заменяется на `chonkie` с `SemanticChunker`,
- весь retrieval-контур сохраняет offline-режим.

Это уже важный инженерный переход: не строить новую систему с нуля, а заменить слабые звенья внутри того же интерфейса.

## Локальный embedding layer

Для embeddings используется `llama.cpp` в серверном режиме и GGUF-модель `jina-embeddings-v4-text-retrieval`.

Практическая ценность такого решения:
- модель специализирована под retrieval,
- не нужны внешние API,
- можно контролировать latency и ресурсы,
- можно работать с Matryoshka embeddings и уменьшать размерность вектора.

Отдельно важна разница между `Query:` и `Passage:`. Для Jina retrieval-модели это не косметика, а часть контракта модели. Если кодировать запросы и документы одинаково, качество retrieval снижается.

## Локальный reranker

Следующий важный шаг, переход от `Reciprocal Rank Fusion` к локальному neural reranker.

Используется `jina-reranker-v3` через `transformers` и `torch`.

Это усиливает систему потому, что теперь финальный порядок кандидатов определяется не только их позицией в двух списках, а semantic соответствием пары:
- запрос,
- кандидатный фрагмент.

Такой переход особенно важен, когда recall уже достаточен, но качество верхних позиций ещё плавает.

## Semantic chunking

Ещё одно слабое место стартовой схемы, чанкирование.

Во второй итерации используется `chonkie` с `SemanticChunker` и overlap refinement.

Это улучшает retrieval по нескольким причинам:
- границы чанков начинают лучше совпадать со смысловыми блоками,
- меньше случайных разрывов внутри тесно связанных фрагментов,
- меньше потери контекста на стыках,
- выше шанс, что retrieval вернёт содержательно цельный кусок.

На практике quality chunking часто влияет на результат не меньше, чем выбор embedding-модели.

## Что в этой схеме действительно сильное

Сильная сторона не в конкретных брендах моделей, а в последовательности развития системы.

Сначала собирается простая, прозрачная и проверяемая архитектура:
- Postgres,
- таблицы,
- пайплайны,
- keyword + vector retrieval,
- baseline fusion.

Потом усиливаются отдельные звенья:
- embedding,
- reranking,
- chunking.

То есть система растёт не через переписывание всего контура, а через замену quality-layer внутри устойчивого каркаса.

Это хороший инженерный паттерн.

## Ограничения

При всей полезности схемы ограничения остаются:
- `transformers` и CUDA добавляют operational complexity,
- локальные модели требуют GPU и памяти,
- обновление документа через remove + recreate грубовато,
- качество retrieval по-прежнему зависит от корпуса и chunking strategy,
- без evaluation нельзя честно утверждать, что новая версия действительно лучше старой.

Иначе говоря, архитектура стала сильнее, но не магической.

## Когда этот паттерн оправдан

Такой стек особенно уместен, когда нужны:
- локальная работа без облачных API,
- объединение keyword и vector retrieval в одном хранилище,
- постепенная эволюция retrieval-контура,
- контроль над operational layer,
- возможность менять модели без разрушения общей архитектуры.

Если задача проще, такой стек может быть тяжёлым.

Если же нужен именно offline retrieval с перспективой роста, это уже выглядит как нормальная база, а не просто демо.

## Вывод

`PostgreSQL + VectorChord` в этом цикле статей ценен не как отдельный трюк, а как цельная дорожная карта для локального hybrid retrieval.

Правильный смысл здесь такой:
1. отделить документы от retrieval-чанков,
2. собрать working pipeline на Postgres,
3. объединить keyword и vector search,
4. потом по одному усилить embedding, reranking и chunking,
5. сохранить общую архитектуру и offline-режим.

В таком виде это уже одна внятная статья про архитектурный паттерн, а не две разрозненные заметки.
