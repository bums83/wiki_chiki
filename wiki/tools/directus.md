# Directus

`Directus` — это платформа, которая накладывает API и админский слой поверх существующей или новой SQL-базы, не заставляя переносить данные в собственную CMS-модель.

Практически это означает, что база остаётся основной системой хранения, а Directus добавляет сверху:
- REST API,
- GraphQL API,
- аутентификацию и доступы,
- админскую панель,
- automation и расширения.

## Что это за класс инструмента

Directus полезно понимать не как обычную CMS, а как database-mirroring backend platform.

Он не навязывает собственную структуру контента как первичную модель. Вместо этого он отражает уже существующую SQL-схему и делает её управляемой через API и UI.

Поэтому он уместен в сценариях, где нужно быстро получить backend layer без написания отдельного CRUD-приложения поверх каждой таблицы.

## Что он даёт

С точки зрения рабочего применения платформа закрывает сразу несколько задач:
- мгновенный API поверх SQL,
- админку для внутренних пользователей,
- управление ролями и доступом,
- работу с файлами и медиа,
- расширяемость через extensions и automation layer.

В результате Directus часто используется как быстрый backend для:
- headless CMS,
- внутренних админок,
- data-driven приложений,
- контентных платформ,
- low-code и no-code operational интерфейсов.

## Почему это может быть выгоднее ручного backend-кода

Обычный путь, написать своё API и отдельно собрать админку. Это даёт максимум контроля, но почти всегда означает повторение типовых задач:
- CRUD для таблиц,
- фильтрация и сортировка,
- роли и права,
- формы редактирования,
- relationship handling,
- файловый слой,
- служебные интерфейсы для команды.

Directus выгоден там, где все эти слои нужны быстро и без желания собирать их вручную заново.

То есть он не заменяет custom backend во всех случаях, но сильно сокращает цену входа для большого класса систем.

## Почему отсутствие vendor lock-in здесь важно

Одна из сильных сторон Directus, работа с чистой SQL-базой.

Это важно потому, что:
- данные остаются в нормальной базе,
- связи и таблицы не прячутся в закрытый runtime-формат,
- можно использовать свои миграции, индексы и SQL-инструменты,
- при отказе от платформы база не превращается в заложника CMS.

Именно поэтому Directus выглядит сильнее многих no-code или headless-платформ, которые фактически требуют жить внутри их собственной модели.

## Поддерживаемые базы и режимы развёртывания

По описанию проекта Directus работает с несколькими SQL-движками, включая:
- PostgreSQL,
- MySQL,
- SQLite,
- MariaDB,
- MS-SQL,
- OracleDB,
- CockroachDB.

Развёртывание возможно:
- локально,
- self-hosted,
- в облаке Directus Cloud.

Это делает платформу удобной и для небольших внутренних систем, и для более серьёзных production-сценариев.

## Админка и no-code слой

Отдельная ценность Directus, админский интерфейс для нетехнических пользователей.

Это полезно в командах, где разработчик не хочет быть постоянным посредником между базой и бизнес-пользователем.

Если редактор, оператор или менеджер может работать через готовую панель, система перестаёт зависеть от постоянных ручных правок со стороны разработчика.

В этом смысле Directus, это не только API-generator, но и operational UI layer над данными.

С инженерной точки зрения Directus стоит рядом с [PostgreSQL + VectorChord]({{ '/wiki/infra/postgresql-vectorchord-hybrid-search' | relative_url }}), но решает другую задачу. Там база используется как основа retrieval и search-инфраструктуры, а здесь как основа управляемого backend и admin-платформы поверх SQL.

## Где он особенно уместен

Directus особенно хорош там, где нужно:
- быстро поднять backend поверх SQL,
- получить API и админку без отдельной разработки,
- оставить данные в своей базе,
- дать бизнесу интерфейс для работы с контентом или справочниками,
- собрать внутренний data-driven tool.

Это делает его близким не к классическим CMS в узком смысле, а к платформам для быстрого построения управляемого backend-слоя.

## Ограничения

При этом важно видеть и границы.

Во-первых, Directus не отменяет потребность в проектировании схемы данных. Плохая база не станет хорошей только из-за красивой панели.

Во-вторых, для сильно специализированной бизнес-логики всё равно могут понадобиться custom endpoints и отдельные сервисы.

В-третьих, лицензия у проекта не классическая permissive open-source, а `BSL 1.1` с дополнительными условиями использования. Это надо учитывать отдельно, особенно для production в более крупных организациях.

## Практический вывод

`Directus` — это сильный инструмент для случаев, когда нужен быстрый backend и админка поверх SQL без потери контроля над самими данными.

Его главная ценность не в том, что он «заменяет backend-разработку вообще», а в том, что он радикально ускоряет типовые backend-задачи там, где архитектура действительно может строиться вокруг существующей базы.
