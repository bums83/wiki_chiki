# Research org code

Идея, появляющаяся вокруг [Autoresearch]({{ '/wiki/llm-agents/autoresearch' | relative_url }}): исследователь всё меньше пишет training code сам и всё больше пишет **код исследовательской организации**.

## Суть

В классическом режиме исследователь:
- меняет Python-код,
- запускает эксперимент,
- смотрит метрики,
- повторяет.

В новой схеме:
- человек задаёт правила, рамки, стиль поиска и критерии через Markdown / prompt layer;
- агент уже внутри этих рамок меняет training code и проводит эксперименты.

## Почему это важно

Это переносит точку оптимизации вверх по стеку:
- с уровня отдельных гиперпараметров,
- на уровень процедуры поиска,
- дисциплины логирования,
- правил принятия изменений,
- разделения ролей между человеком и агентом.

Проще говоря, объект инженерии становится не только модель, но и **сама исследовательская машина**.

## Практическая форма

В `autoresearch` эту роль играет `program.md`.

Это не просто prompt. Это лёгкий operational spec:
- что агенту можно менять,
- как он должен интерпретировать улучшение,
- как оформлять шаги,
- когда считать эксперимент удачным.

## Прикладной аналог

В software engineering близкую логику можно увидеть в [Вайб-кодинге]({{ '/wiki/llm-agents/vibe-coding' | relative_url }}): там человек тоже перестаёт быть только исполнителем и начинает проектировать рамки, playbook-и и порядок работы агента.

Разница в области применения:
- `research org code` управляет исследовательским loop;
- `vibe coding` управляет прикладной разработкой, рефакторингом и интеграциями.

## Вывод

`Research org code` — полезная рамка для понимания agentic tooling. Настоящая ценность появляется не тогда, когда агент просто «умеет кодить», а когда человек умеет спроектировать для него среду поиска.
