# Autoresearch

`autoresearch` — минималистичный framework Андрея Карпаты для автономного LLM research на одном GPU. Идея проста: не исследователь руками гоняет десятки мелких экспериментов, а агент делает это сам в коротком цикле, пока человек спит.

## Что это делает

Агент работает по loop:
1. меняет training code,
2. запускает короткое обучение,
3. сравнивает результат по метрике,
4. сохраняет удачное изменение или откатывает неудачное,
5. повторяет цикл.

Базовый цикл фиксирован по времени: **5 минут на эксперимент**. Это даёт сопоставимость результатов между запусками.

## Архитектура

Проект специально ужат до трёх главных файлов:

- `prepare.py` — подготовка данных, токенизатор, runtime-утилиты, не меняется;
- `train.py` — модель, оптимизатор и training loop, это рабочая поверхность агента;
- `program.md` — instructions/context, которые задаёт человек.

Ключевое разделение ролей:
- **человек программирует исследовательскую организацию** через `program.md`;
- **агент программирует training system** через `train.py`.

Это важный сдвиг. Исследователь всё меньше правит код напрямую и всё больше проектирует operational envelope, в котором агент может безопасно и продуктивно искать улучшения.

## Почему это интересно

### 1. Один файл для правок
Ограниченный search space делает поведение агента обозримым. Diff-ы можно читать и ревьюить без хаоса по всему репозиторию.

### 2. Фиксированный time budget
Если каждый прогон длится одинаково, сравнение экспериментов становится честнее. Система ищет не «лучшую модель вообще», а лучшую конфигурацию в данном вычислительном бюджете.

### 3. Research as operations
`program.md` здесь по сути становится кодом исследовательской организации. Это уже не просто prompt, а policy layer: что агенту разрешено трогать, как он должен мыслить, как фиксировать результат, когда принимать изменение.

### 4. Ночной throughput
README заявляет ориентир порядка **~12 экспериментов в час** и **~100 экспериментов за ночь**. Это делает проект интересным не только как toy-demo, но и как практический personal research loop.

## Метрика

Основная метрика — `val_bpb` (validation bits per byte). Ниже — лучше. В README подчёркивается, что такая метрика удобна для сравнения разных архитектурных решений и не зависит напрямую от размера словаря так, как зависели бы другие показатели.

## Что показали первые результаты

Из публичных комментариев Карпаты следует, что `autoresearch` уже находил **наборы улучшений**, а не только отдельные твики:

- за ~2 дня система нашла около 20 улучшений на depth=12 модели;
- позже эти улучшения были отдельно проверены;
- они оказались additive;
- они перенеслись и на более крупную depth=24 модель.

Это важное наблюдение. Значит, агентный поиск может находить изменения, которые живут не только в узкой локальной точке, а масштабируются дальше.

## Ограничения

Проект нарочно узкий:
- один GPU;
- в README основной reference setup привязан к NVIDIA;
- без distributed training;
- без сложных конфигов;
- без широкой аппаратной совместимости из коробки.

Именно в этом его сила. Это не универсальная research platform, а компактный полигон для быстрых автономных итераций.

## Для кого это

- для исследователей, которым нужен personal overnight loop;
- для инженеров, которые хотят проверять architecture / optimizer / batch / tokenizer tweaks без ручного микроменеджмента;
- для тех, кто изучает новую модель взаимодействия «human writes the org, agent writes the code».

## Экосистема вокруг autoresearch

`autoresearch` лучше понимать не изолированно, а как часть более широкого кластера:

- [[Nanochat]] — базовый training harness;
- [[Validation bits per byte]] — быстрая метрика обратной связи;
- [[Overnight experimentation]] — operational pattern пакетного ночного поиска;
- [[Asynchronous research swarms]] — следующий шаг от одного агента к распределённому research community.

## Практический вывод

`autoresearch` важен не столько как ещё один training repo, сколько как рабочий прототип нового интерфейса к research:

**человек задаёт систему поиска, агент исполняет и накапливает улучшения.**

Если эта модель закрепится, следующим объектом оптимизации станет уже не только `train.py`, а сама конструкция research-процесса.
